# Part II: System Architecture and Design

## 2.1. Architectural Blueprint (The Decentralized Stack)

EchoinWhispr is a fully serverless dApp. The architecture is defined by five distinct layers, replacing all traditional back-end components.

| Layer | Service / Technology | Role in EchoinWhispr |
|-------|----------------------|----------------------|
| Identity & Auth | Hedera Accounts / Wallet Connect | Non-custodial login. The Hedera Account ID is the user's permanent, global identifier. |
| State & Logic | Hedera Smart Contract Service (Solidity EVM) | Manages all mutable user data: [`UserProfile`](Back-End/smart-contracts/EchoinWhispr.sol:23) struct (Persona, [`mailHash`](Back-End/smart-contracts/EchoinWhispr.sol:28), [`publicKey`](Back-End/smart-contracts/EchoinWhispr.sol:27)), [`subscriptionEnds`](Back-End/smart-contracts/EchoinWhispr.sol:29) timestamp. Executes logic (matching, subscription). |
| Data Storage | IPFS (InterPlanetary File System) / Filecoin | Decentralized, content-addressed storage for:<br>1. Raw image files (unencrypted).<br>2. The encrypted Message Object (ciphertext). |
| Consensus & Routing | Hedera Consensus Service (HCS) | Provides an immutable, timestamped log for "Conversation Messages." HCS topic messages serve as a decentralized, real-time message broker, routing CIDs to anonymous hashes. |
| Data Reading | Hedera Mirror Nodes (JSON-RPC Relay) | Used by the client to query the Smart Contract state (for search, profile loading) and to subscribe to the HCS Conversation Topic for real-time message alerts. |

## 2.2. On-Chain vs. Off-Chain Data Strategy

This data model is optimized for cost, performance, and privacy.

### ON-CHAIN (Hedera Smart Contract):

- **What:** Small, critical, state-based data.
- **Data:** The [`UserProfile`](Back-End/smart-contracts/EchoinWhispr.sol:23) struct, containing [`career`](Back-End/smart-contracts/EchoinWhispr.sol:24), [`interests`](Back-End/smart-contracts/EchoinWhispr.sol:25)[], [`currentMood`](Back-End/smart-contracts/EchoinWhispr.sol:26), [`publicKey`](Back-End/smart-contracts/EchoinWhispr.sol:27), [`mailHash`](Back-End/smart-contracts/EchoinWhispr.sol:28), and [`subscriptionEnds`](Back-End/smart-contracts/EchoinWhispr.sol:29).
- **Why:** This data needs to be mutable (editable) and queryable (for search/matching). Storing it in a contract is more gas-efficient than HCS for frequent edits and complex reads.

### ON-CHAIN (Hedera Consensus Service):

- **What:** Immutable, timestamped event logs.
- **Data:** The "Conversation Message" notification. This message is a JSON payload (see 4.6) containing routing hashes (to_hash, reply_hash), the sender's public key (from_key), and a pointer to the off-chain data (cid).
- **Why:** HCS is extremely low-cost and high-throughput, perfect for a high-volume messaging event bus. It provides the routing pointer and the data location without storing the heavy, encrypted data itself.

### OFF-CHAIN (IPFS / Filecoin):

- **What:** Large, private, or rich-media data.
- **Data 1:** The raw image file (e.g., design.jpg). This file is not encrypted but is stored at a content-addressed hash (image_cid). Its privacy is protected because the image_cid is only ever revealed inside an encrypted message.
- **Data 2:** The Message Object (see 3.3) which contains the text and the image_cid. This entire object is encrypted and stored on IPFS, creating a main_cid.
- **Why:** Storing large data on-chain is prohibitively expensive. This nested-CID model ensures the message (and the link to the image) is private, while the image is stored efficiently.
## 2.3. Technology Stack

- **Client (Frontend):** Web Application (React, Vue, or Svelte)
- **Wallet Interface:** WalletConnect v2.0, compatible with Hedera wallets (HashPack).
- **Core SDKs:**
  - hedera-sdk-js: For all interactions with HCS, token transfers, and native (non-EVM) operations.
  - ethers.js: For interfacing with the EVM via the JSON-RPC relay (contract calls).
- **Encryption:** ecies-js (or similar ECIES-compatible library) for client-side elliptic curve encryption.
- **Storage Interface:** ipfs-http-client (or equivalent) to interact with an IPFS node (e.g., Infura, Pinata, or a local node).
