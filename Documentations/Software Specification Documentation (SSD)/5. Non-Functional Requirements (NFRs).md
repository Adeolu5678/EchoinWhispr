# Part V: Non-Functional Requirements (NFRs)

## 5.1. Security & Anonymity:

- **No Plaintext:** At no point shall unencrypted message content (text or image links) touch any part of the Hedera network (contract, HCS).
- **No Account IDs:** The HCS log never contains a user's Hedera Account ID. It only contains un-linkable hashes and public keys.
- **E2E-Encryption:** All message content is encrypted client-side using ECIES.
- **Unlinkable Conversations:** A user can initiate two separate conversations with the same person, and that person will see them as two separate conversations from two different anonymous hashes (as a new reply_hash is generated for each).

## 5.2. Performance & Scalability:

- **Client-Side Load:** The client application (browser) assumes responsibility for search filtering and HCS stream filtering. This must be optimized to handle a large HCS topic stream.
- **HCS Throughput:** The system must leverage HCS's 10,000+ TPS for message routing.
- **Image Loading:** Image-heavy conversations will require multiple IPFS fetches. The client must implement lazy-loading for images to ensure a smooth UI.

## 5.3. Usability & Accessibility (UX):

- **Web2 Familiarity:** The dApp's UI must abstract all Web3 complexity. The user should feel like they are using a standard web app.
- **Transaction Obfuscation:** Wallet prompts must be clear and descriptive (e.g., "Sign to create your profile" or "Sign to send your Whisper").

## 5.4. Cost & Gas Efficiency:

- **Read-Heavy:** Architecture is designed for free Mirror Node queries (for search, profile loading).
- **Write-Light:** EVM writes are limited to one-time registration, infrequent persona edits, and periodic subscriptions.
- **Message Cost:** Message sending (HCS) is optimized for the lowest possible cost per-message on the Hedera network.

## 5.5. Decentralization:

- **No Central Server:** The application logic must not rely on any centralized, developer-owned server for core functionality.
- **Full Stack:** The entire stack (Identity, Logic, Storage, Routing) is decentralized as specified in Part II.
