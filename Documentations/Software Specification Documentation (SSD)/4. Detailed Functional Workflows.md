## 4.6. Workflow: Anonymous Conversation Flow (with Images)

This is the most critical and complex workflow in the system. It enables two-way, persistent, anonymous, end-to-end encrypted conversations with rich media.

### Definitions:

- **mailHash:** A user's permanent, public anonymous "mailbox" hash. Stored in the smart contract. Used only to receive the first message of a new conversation.
- **reply_hash:** A temporary, random hash, generated by a message sender. This is the hash the recipient will use as the to_hash for their reply. It's a one-time-use "reply-to" address.
- **convo_id:** A client-side-only ID (e.g., UUID) to group messages into a single chat window.

### HCS Message Payload (JSON):

This is the unencrypted message submitted to the HCS Topic. Its sole purpose is routing and key exchange.

```json
{
  "to_hash": "...",       // bytes32: The hash this message is being sent to.
  "reply_hash": "...",    // bytes32: The new random hash the recipient must use to reply.
  "from_key": "...",      // bytes: The sender's public encryption key.
  "cid": "...",           // string: The IPFS hash of the *encrypted message object*.
  "convo_id": "..."     // string (UUID): The client-side thread ID.
}
```


### Client-Side Filtering (The "Mailbox"):

Every client, upon login, subscribes to the entire HCS Whisper Topic via a Hedera Mirror Node. It filters the full stream for messages where:

- to_hash == my_mailHash (This catches new conversation invitations).
- to_hash is in my_list_of_active_reply_hashes (This catches replies to conversations I'm in).

### A. Initiating a Conversation (User A -> User B)

- **Trigger:** User A (Sender) clicks "Send Whisper" on User B's (Recipient) profile.
- **Get Data:** Client A queries the contract for User B's info:
  - B_mailHash = [getMailHash()](Back-End/smart-contracts/EchoinWhispr.sol:212)(B_Account_ID)
  - B_publicKey = [getPublicKey()](Back-End/smart-contracts/EchoinWhispr.sol:204)(B_Account_ID)
- **Generate Data:** Client A generates:
  - new_convo_id: "uuid-1234"
  - my_reply_hash_A1: A new, random bytes32 hash ("REPLY-A1").
  - my_publicKey_A: (Client A already knows this).
- **Store Reply Hash:** Client A adds "REPLY-A1" to its local list of active reply hashes to filter for.
- **Encrypt & Upload (Rich Media Workflow):**
  - **5a. (If Image):** User A attaches design.png. Client A uploads this to IPFS, receiving image_cid_1. (If no image, this is null).
  - **5b. (Compose):** Client A creates the Message Object (Plaintext):
    ```json
    { "type": "text/image", "text_content": "Hi! I saw your profile.", "image_cid": "image_cid_1" }
    ```
  - **5c. (Encrypt):** Client A encrypts this JSON string with B_publicKey to create ciphertext_1.
  - **5d. (Upload):** Client A uploads ciphertext_1 to IPFS, receiving main_cid_1.
- **Format HCS Message:**
  ```json
  {
    "to_hash": "B_mailHash",      // Send to B's permanent, public mailbox
    "reply_hash": "REPLY-A1",     // The address B must use to reply
    "from_key": "my_publicKey_A", // B needs this to encrypt their reply
    "cid": "main_cid_1",          // Pointer to the encrypted message
    "convo_id": "uuid-1234"       // Client-side thread identifier
  }
  ```


- **Submit:** Client A signs and submits this payload to the HCS Topic.

### B. Receiving a New Conversation (User B)

- **Filtering:** Client B's filter is listening. It finds a message where to_hash == B's_mailHash.
- **Ingest:** Client B ingests the HCS message.
- **Decrypt:**
  - Client B fetches main_cid_1 from IPFS to get ciphertext_1.
  - It decrypts ciphertext_1 with its own private key to reveal the Message Object (Plaintext) JSON string.
- **Parse & Display:**
  - Client B parses the JSON.
  - It displays the text_content ("Hi! I saw your profile.") in a new chat window (using convo_id).
  - It checks image_cid. If not null, it then fetches the image from image_cid_1 on IPFS and displays it in the chat window (ideally with lazy-loading).
- **Store Reply Data:** Client B stores this data, associated with "uuid-1234":
  - reply_to_hash_for_B: "REPLY-A1" (The reply_hash from the message).
  - reply_to_key_for_B: "my_publicKey_A" (The from_key from the message).

### C. Replying to the Conversation (User B -> User A)

- **Trigger:** User B types a reply (and optionally attaches reply_image.png) in the "uuid-1234" conversation.
- **Get Data:** Client B retrieves the stored data for this thread:
  - target_hash: "REPLY-A1"
  - target_key: "my_publicKey_A"
- **Generate Data:** Client B generates its own new reply hash:
  - my_reply_hash_B1: A new, random bytes32 hash ("REPLY-B1").
- **Store Reply Hash:** Client B adds "REPLY-B1" to its filter list.
- **Encrypt & Upload (Rich Media Workflow):**
  - **5a. (If Image):** Client B uploads reply_image.png to IPFS, receiving image_cid_2.
  - **5b. (Compose):**
    ```json
    { "type": "text/image", "text_content": "Great design!", "image_cid": "image_cid_2" }
    ```
  - **5c. (Encrypt):** Client B encrypts this JSON string with target_key ("my_publicKey_A") to create ciphertext_2.
  - **5d. (Upload):** Client B uploads ciphertext_2 to IPFS, receiving main_cid_2.
- **Format HCS Message:**
  ```json
  {
    "to_hash": "REPLY-A1",      // Sending to A's temporary hash
    "reply_hash": "REPLY-B1",     // A's *next* reply will come here
    "from_key": "my_publicKey_B", // A needs this to encrypt their *next* reply
    "cid": "main_cid_2",
    "convo_id": "uuid-1234"    // Same conversation ID
  }
  ```


- **Submit:** Client B signs and submits this to the HCS Topic.

### D. Receiving the Reply (User A)

- **Filtering:** Client A's filter is listening. It finds a message where to_hash == "REPLY-A1".
- **Ingest & Decrypt:** Client A fetches main_cid_2, decrypts it with its private key, and gets the Message Object (Plaintext) JSON.
- **Parse & Display:** Client A parses the JSON, displays the text ("Great design!"), and fetches/displays the image from image_cid_2.
- **Update Reply Data:** Client A updates the stored data for this thread. This is the critical "hot-potato" pass:
  - The new reply_to_hash_for_A is now "REPLY-B1".
  - The new reply_to_key_for_A is now "my_publicKey_B".
- **Cleanup (Optional):** Client A can now remove "REPLY-A1" from its filter list, as it has been "used" and replaced by "REPLY-B1".

This "hot-potato" passing of new reply hashes ensures each message routes correctly, remains E2E-encrypted, and supports rich media, all while preserving 100% anonymity and unlinkability.
