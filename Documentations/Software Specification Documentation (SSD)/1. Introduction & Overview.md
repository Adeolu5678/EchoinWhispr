# Part I: Foundational Project Specification

## 1.1. Introduction and Document Purpose

This document is the definitive Software Specification Document (SSD) for the EchoinWhispr application. It serves as the single source of truth for all stakeholders.

**Project Vision:** To create a revolutionary social application that bridges the "Great Social Divide." EchoinWhispr is a "social equalizer" designed to connect socially isolated individuals with a broader user base. It is a new social fabric where merit, skill, and personality are the only currency that matters.

**Document Purpose:** To provide a comprehensive guide for the project's MVP lifecycle, detailing the system's purpose, architecture, functional requirements, and non-functional constraints.

## 1.2. System Overview & Core Value Proposition

EchoinWhispr is a modern web application built on Next.js and Convex. Users connect via standard authentication (Clerk), create a "Persona" based on their skills and interests, and connect with others through an anonymous messaging system.

**Core Value Proposition: Trust & Equality by Design.**

- **Trust:** Secure, authenticated access with transparent platform rules.
- **Equality:** The system is anonymous. It removes real-world barriers like status, finance, and popularity, allowing users to connect based on shared interests and professional merit.

## 1.3. User Personas

- **Persona 1: The "Tech-Savvy" User**
  - **Description:** A technically proficient innovator or developer.
  - **Problem:** Socially isolated within the technical sphere.
  - **Goal:** To find and connect with like-minded individuals without the friction of traditional social media.

- **Persona 2: The "Mainstream" User**
  - **Description:** A professional, creative, or student who values privacy and meaningful connections.
  - **Problem:** Intimidated by complex platforms or superficial social networks.
  - **Goal:** To experience the benefits of anonymous connection through an intuitive, familiar interface.

- **Persona 3: The "Merit-Focused" User**
  - **Description:** Any individual who feels blocked by real-world social hierarchies.
  - **Problem:** Connection ability dictated by status rather than skill.
  - **Goal:** To initiate conversations based purely on merit, shared career fields, or common interests.

- **Persona 4: The "Administrator" User**
  - **Description:** A trusted community member elevated to admin status.
  - **Problem:** Need visibility into platform activity for moderation and trust/safety.
  - **Goal:** To monitor anonymous whispers with full sender/recipient details and manage admin promotions.

## 1.4. User Journeys (Epics & Use Cases)

### Use Case 1: New User Registration & Persona Creation

- **Actor:** New User.
- **Trigger:** User visits the app and clicks "Sign Up."
- **Steps:**
  1. User authenticates via Clerk (Email, Google, etc.).
  2. App checks if a user profile exists in Convex.
  3. If not, displays "Create Your Persona" form.
  4. User fills in Career, Interests, and Mood.
  5. User clicks "Create Persona."
  6. Client calls `mutation users.register`.
- **Post-condition:** User is logged in and searchable.

### Use Case 2: Persona Editing

- **Actor:** Existing User.
- **Trigger:** User navigates to "Profile."
- **Steps:**
  1. App displays current Persona data.
  2. User modifies fields.
  3. User clicks "Save Changes."
  4. Client calls `mutation users.updatePersona`.
- **Post-condition:** User profile is updated in the database.

### Use Case 3: Subscribing to Premium

- **Actor:** Existing User (Free Tier).
- **Trigger:** User clicks "Go Premium."
- **Steps:**
  1. User selects a plan.
  2. App redirects to Stripe Checkout.
  3. User completes payment.
  4. Stripe webhook updates user status in Convex.
- **Post-condition:** User has premium access.

### Use Case 4: Search & Filter

- **Actor:** Existing User.
- **Trigger:** User navigates to "Search."
- **Steps:**
  1. User enters search criteria (Career, Interests).
  2. Client calls `query users.search`.
  3. App displays matching anonymous personas.
  4. User can click "Send Whisper" to initiate conversation.
- **Post-condition:** User sees relevant profiles.

### Use Case 5: Mood Matching

- **Actor:** Existing User.
- **Trigger:** User uses "Mood Match."
- **Steps:**
  1. Client calls `query users.findMoodMatch` with current mood.
  2. Backend finds a random matching user.
  3. App displays the match.
- **Post-condition:** User is presented with a match.

### Use Case 6: Sending & Receiving Messages

See Part IV for detailed workflows.

### Use Case 7: Requesting Admin Privileges

- **Actor:** Existing User.
- **Trigger:** User navigates to Settings and clicks "Request Admin Access."
- **Steps:**
  1. User fills out reason for admin request.
  2. Client calls `mutation admin.requestAdminPromotion`.
  3. Request is stored with "pending" status.
  4. Super Admin receives notification of pending request.
- **Post-condition:** Request is queued for review.

### Use Case 8: Admin Whisper Monitoring

- **Actor:** Admin User.
- **Trigger:** Admin navigates to Admin Dashboard.
- **Steps:**
  1. Client calls `query admin.getAllWhispers` with pagination.
  2. Dashboard displays all whispers with sender/recipient usernames.
  3. Admin clicks "Expand" on a whisper row.
  4. Client calls `query admin.getWhisperDetails` for full info.
  5. Panel reveals complete sender/recipient profiles and message details.
- **Post-condition:** Admin has visibility into platform communications.

## 1.5. Project Scope

### In Scope (MVP Features):

- **MVP 1:** Authentication (Clerk v6).
- **MVP 2:** Profile Management (Convex).
  - Persona Creation & Editing.
  - New "Resonance" fields (Life Phase, Mentorship).
- **MVP 3:** Subscription Service (Framework for Stripe).
- **MVP 4:** Search & Discovery.
- **MVP 5:** Mood Matching Logic.
- **MVP 6:** Real-time Messaging (Convex).
  - Anonymous Whispers -> Two-way Conversations.
- **MVP 7:** Rich Media Support (Convex Storage, Prompt Widgets).
- **MVP 8:** Admin System.
  - Admin privilege request/grant flow.
  - Admin dashboard with whisper monitoring.
  - Role-based access control (admin, super_admin).
- **MVP 9:** Mobile-Responsive Web Design (PWA-like).

### Out of Scope (Future Iterations):

- "Connect for Romance" Pillar.
- Advanced Verification.
- Native Mobile Apps (React Native/Flutter standalone).
