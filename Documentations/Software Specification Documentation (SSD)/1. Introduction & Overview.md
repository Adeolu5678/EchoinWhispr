# Part I: Foundational Project Specification

## 1.1. Introduction and Document Purpose

This document is the definitive Software Specification Document (SSD) for the EchoinWhispr dApp. It serves as the single source of truth for all stakeholders, including the development team, designers, and project managers.

**Project Vision:** To create a revolutionary, fully decentralized social application that bridges the "Great Social Divide." EchoinWhispr is a "social equalizer" built on Hedera, designed to connect the socially isolated Web3 community with the intimidation-barrier-facing Web2 user base. It is a new social fabric where merit, skill, and personality are the only currency that matters.

**Document Purpose:** To provide a comprehensive guide for the project's entire MVP lifecycle, detailing the system's purpose, architecture, functional requirements, and non-functional constraints. It is intended to eliminate ambiguity and ensure all development work aligns with the project's strategic goals.

## 1.2. System Overview & Core Value Proposition

EchoinWhispr is a fully serverless decentralized application (dApp) that replaces traditional back-end infrastructure with Hedera's suite of services. Users connect anonymously via their Web3 wallets, create a "Persona" based on their skills and interests, and connect with others through an end-to-end encrypted, anonymous messaging system.

**Core Value Proposition: Trust & Equality by Design.**

- **Trust:** All logic is transparent and executed by immutable smart contracts. Message routing is handled by the tamper-proof Hedera Consensus Service (HCS).
- **Equality:** The system is anonymous. It removes real-world barriers like status, finance, and popularity, allowing users to connect based on shared interests and professional merit.
## 1.3. User Personas

- **Persona 1: The "Web3 Echo Chamber" User**
  - **Description:** A technically proficient Web3 innovator, developer, or enthusiast.
  - **Problem:** Socially isolated within the technical sphere; finds it difficult to connect with mainstream users or professionals outside their niche.
  - **Goal:** To find and connect with like-minded individuals, potential collaborators, or a broader audience without the friction of traditional social media.

- **Persona 2: The "Web2 Intimidation" User**
  - **Description:** A mainstream user (professional, creative, student) who is curious about Web3 but intimidated by its complexity (wallets, gas fees, DIDs).
  - **Problem:** Feels there is a high barrier to entry for decentralized platforms.
  - **Goal:** To experience the benefits of Web3 (anonymity, equality) through an intuitive, familiar, and "Web2-friendly" user interface.

- **Persona 3: The "Real-World Wall" User**
  - **Description:** Any individual (professional, artist, entrepreneur) who feels blocked by real-world social and professional hierarchies.
  - **Problem:** Their ability to connect is often dictated by status, popularity, or financial standing, not their actual skills or personality.
  - **Goal:** To bypass these "walls" and initiate conversations based purely on merit, shared career fields, or common interests.
## 1.4. User Journeys (Epics & Use Cases)

### Use Case 1: New User Registration & Persona Creation

- **Actor:** New User (any Persona).
- **Trigger:** User visits the dApp for the first time and clicks "Connect Wallet."
- **Steps:**
  1. The dApp prompts the user to connect their Hedera wallet (e.g., HashPack) via WalletConnect.
  2. User approves the connection. The dApp reads their Hedera Account ID (e.g., 0.0.12345).
  3. The dApp queries the EchoinWhispr.sol contract to see if a profile exists for this address.
  4. The contract returns false (or empty data).
  5. The dApp displays the "Create Your Persona" registration form.
  6. User must fill in the following fields:
     - Career (string, e.g., "Blockchain Developer")
     - Interests (list of strings, e.g., ["DeFi", "Hiking", "Jazz"])
     - Current Mood (string, e.g., "Creative")
  7. The user's client generates a public/private key pair locally for ECIES message encryption.
  8. The client also generates a mailHash. This is a permanent, anonymous "mailbox" hash (e.g., keccak256(abi.encodePacked(msg.sender, "some_secret_salt_from_wallet_signature"))).
  9. User clicks "Create Persona."
  10. The client calls registerUser(string _career, string[] _interests, string _mood, bytes _publicKey, bytes32 _mailHash).
  11. User signs the transaction in their wallet.
- **Post-condition:** The user is logged in, their anonymous persona is now searchable, and their mailHash is on-chain.
### Use Case 2: Persona Editing

- **Actor:** Existing User.
- **Trigger:** User navigates to their "Profile" or "Settings" page.
- **Steps:**
  1. The dApp displays the user's current Persona data (Career, Interests, Current Mood).
  2. User modifies one or more fields.
  3. User clicks "Save Changes."
  4. The client calls the editUserPersona(string _career, string[] _interests, string _mood) function.
  5. User signs the transaction.
- **Post-condition:** The user's persona is updated on-chain. (Note: publicKey and mailHash are permanent and cannot be edited).

### Use Case 3: Subscribing to Premium (HBAR & HTS)

- **Actor:** Existing User (Free Tier).
- **Trigger:** User clicks "Go Premium."
- **HBAR Payment:**
  1. Client calls the subscribeWithHBAR() payable function, sending the required hbarSubscriptionPrice as msg.value.
  2. User signs the transaction.
- **HTS Token Payment:**
  1. **Step 4a (Approve):** User first clicks "Approve." The client calls the HTS token's approve(EchoinWhispr.sol_address, htsSubscriptionPrice) function. User signs.
  2. **Step 4b (Subscribe):** Once approved, the user clicks "Subscribe." The client calls subscribeWithHTS(). User signs. The contract then executes transferFrom.
- **Post-condition:** The subscriptionEnds timestamp for msg.sender is updated to block.timestamp + 30 days (or adds 30 days to an existing subscription).
### Use Case 4: Search & Filter (Career/Interests)

- **Actor:** Existing User.
- **Trigger:** User navigates to the "Search" page.
- **Steps:**
  1. The dApp provides filter criteria: a text box for Career and a multi-select or text box for Interests.
  2. User enters "Solidity Developer" and "NFTs."
  3. The client queries a Hedera Mirror Node (JSON-RPC Relay) to fetch all user personas from the smart contract (likely via the paginated getUsers function).
  4. The client-side application performs the filtering, iterating through all profiles to find matches for Career == "Solidity Developer" AND Interests contains "NFTs".
  5. The dApp displays a list of matching anonymous personas. Each result shows:
     - Career: "Solidity Developer"
     - Interests: ["DeFi", "NFTs", "Security"]
     - Current Mood: "Focused"
     - (No name, no Hedera Account ID is displayed to the user)
  6. Alongside each result is a "Send Whisper" button.
- **Post-condition:** User is presented with a list of anonymous, relevant users to contact.

### Use Case 5: Mood Matching

- **Actor:** Existing User.
- **Trigger:** User navigates to the "Mood Match" feature.
- **Steps:**
  1. Client calls the [findMoodMatch()](Back-End/smart-contracts/EchoinWhispr.sol:221)(string _mood) function, passing in the user's own mood.
  2. The smart contract executes pseudorandom logic and returns a single matching address.
  3. The client fetches that address's persona and displays it.
- **Post-condition:** User is presented with a random user who shares their current mood.

## 1.5. Project Scope

### In Scope (MVP Features):

- **MVP 1:** Decentralized Authentication (Hedera Wallet Connect).
- **MVP 2:** Profile & State Management (EVM Smart Contract).
  - Includes Persona Creation (Career, Interests, Mood).
  - Includes Persona Editing.
  - Includes mailHash generation for anonymous messaging.
- **MVP 3:** Subscription Service.
  - Includes payment via HBAR.
  - Includes payment via a specified HTS Token.
- **MVP 4:** Search & Discovery.
  - Includes search by Career.
  - Includes search by Interests.
- **MVP 5:** Mood Matching Logic (Smart Contract function).
- **MVP 6:** Conversation Initiation (Whisper).
- **MVP 7:** Conversation Logging & Routing (HCS Topic).
- **MVP 8:** Two-Way Anonymous Conversations (Receiving & Replying).
- **MVP 9:** Rich Media Support (Ability to send and receive encrypted images).

### Out of Scope (Future Iterations):

- "Connect for Romance" Pillar: The "Persona Swiping" feature is explicitly deferred.
- **POST 1:** Persona Verification (DIDs/VCs).
- **POST 2:** "Echo" Function (Optional De-anonymization).
- **POST 3:** Tokenized Tipping (HTS).
- **POST 4:** Immutable Audit Log (HCS log for matches/events).
- **POST 5:** Decentralized Governance (DAO).
